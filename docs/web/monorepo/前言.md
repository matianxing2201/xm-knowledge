# Monorepo 前言

# PNPM

pnpm 是一个快速、节省磁盘空间的 npm 包管理工具，以中心化思想解决依赖复用问题具。它有以下特点

## 节省磁盘空间

### 全局内容寻址存储

- 所有下载的依赖包统一存储在 `~/.pnpm-store`（默认路径），使用 硬链接 到各项目的 `node_modules`。
- 相同版本的包全局仅存一份，跨项目共享。
- 示例： 如果10 个项目都用 `lodash@4.17.21`，磁盘仅占 1 份空间。

### 依赖复用机制

- 依赖解析时优先匹配 `store` 中的已有版本，避免重复下载。
- 与 npm/yarn 的本地缓存不同，pnpm 直接硬链接到项目，省去拷贝开销。

## 提高安装速度

### 依赖预筛选（Resolution Stage）

- 通过 `store` 比对已存在的依赖，跳过已下载包的网络请求。
- 场景： 当切换 Git 分支时，重复依赖无需重新下载。

### 并行文件硬连接

- 文件链接操作比拷贝快数十倍，且并行处理依赖项。
- 数据对比： 安装 `vite` + `react` 项目：
  - npm：12s
  - pnpm：3s

![安装速度对比 1](/images/web/monorepo/web-monorepo-05.png)

![安装速度对比 2](/images/web/monorepo/web-monorepo-06.png)

### 依赖层级扁平化处理(非目录结构扁平)

- 通过符号链接组织嵌套依赖，减少文件系统遍历深度。

## 非扁平化 `node_modules` 结构

目录机构特点如下

```bash
node_modules/
├── .pnpm/                # 所有依赖的硬链接和元数据
│   ├── lodash@4.17.21/
│   └── react@18.2.0/
├── vite → .pnpm/vite@4.0.0/node_modules/vite/  # 符号链接
└── react → .pnpm/react@18.2.0/node_modules/react/
```

## 核心优势

1. 依赖隔离
   - 每个包仅在自身作用域访问直接依赖，避免 幽灵依赖（Phantom Dependencies）。
   - 示例： 项目未显式安装 `ansi-regex`，但 npm/yarn 提升依赖后可能被访问到；pnpm 则严格隔离。
2. 确定性依赖树
   - 依赖版本冲突时，不同版本的包可共存且路径明确。
   - 场景： A 包依赖 `lodash@4.0.0`，B 包依赖 `lodash@4.17.21`，两者独立存在。
3. 兼容性更强
   - 避免扁平化结构导致的 依赖提升冲突（如 `vue 2/3` 被错误提升到根目录）。

## 总结对比

| 特点                | npm/yarn          | pnpm               |
| ------------------- | ----------------- | ------------------ |
| 磁盘占用            | 高（重复存储）    | 低（硬链接共享）   |
| 安装速度            | 较慢（网络+拷贝） | 快（链接+并行）    |
| `node_modules` 结构 | 扁平化（易冲突）  | 嵌套隔离（确定性） |
| 安全性              | 幽灵依赖风险      | 严格依赖访问       |

# Monorepo(Monolithic Repository)

## 什么是monorepo

Monorepo 是一种工程化开发管理模式，它允许在一个仓库中管理多个项目或模块。

## 优势

- 依赖共享：多个项目可以共享相同的依赖，减少重复安装和版本冲突。
- 版本控制：所有项目都在同一个仓库中，版本控制更加简单，便于统一管理和发布。
- 构建与部署：可以统一构建和部署流程，减少配置的复杂性。
- 代码复用：不同项目之间可以复用代码，减少重复开发。
- 团队协作：团队成员可以更方便地协作，减少跨仓库的沟通成本。

## 应用场景

- 项目的子项目架构：适用于大型项目，将不同的功能模块拆分为子项目，便于管理和维护。
- 工具分类封装：将常用的工具函数、组件库等封装为独立的模块，便于在不同项目中复用。
- 功能切片：对于复杂应用，可以将不同的功能切片为独立的模块，便于开发和测试

## repository项目管理的发展历程

### 单体仓库单应用模式 (Monolith)：

一个仓库只管理一个单一的应用或项目，所有代码和依赖都集中在一个仓库中。

**优点**：

- **简单易用**：项目结构清晰，适合快速启动和开发。
- **部署方便**：只有一个应用，构建和部署流程简单。
- **调试方便**：所有代码在一个仓库中，调试和测试较为直接。

**缺点**：

- **难以扩展**：随着项目功能增加，代码库会变得臃肿，难以维护。
- **耦合度高**：所有功能模块耦合在一起，修改一个模块可能会影响其他模块。
- **团队协作困难**：随着团队规模扩大，多人同时开发一个仓库容易引发冲突。
- **技术栈单一**：难以在同一仓库中使用多种技术栈。

### 多仓库多应用模式（Multirepo）

多个应用分配给多个对应的仓库进行管理

**优点**：

- **模块化**：每个应用或模块独立管理，职责清晰。
- **技术栈灵活**：不同仓库可以使用不同的技术栈，适合多技术栈的项目。
- **独立部署**：每个应用可以独立构建和部署，灵活性高。

**缺点**：

- **依赖管理复杂**：多个仓库之间的依赖关系难以管理，容易出现版本冲突。
- **代码共享困难**：不同仓库之间的代码复用性差，需要重复开发。
- **协作成本高**：跨仓库的协作和版本控制复杂，沟通成本高。
- **构建效率低**：每个仓库需要独立构建，整体构建时间较长。

### 单仓库多应用模式 - Monorepo

一个仓库管理多个应用或项目，所有代码和依赖集中在一个仓库中，但通过工具（如 Lerna、Nx、Rush）进行模块化管理。

**优点**：

- **依赖共享**：多个项目可以共享相同的依赖，减少重复安装和版本冲突。
- **代码复用**：不同项目之间可以复用代码，减少重复开发。
- **统一版本控制**：所有项目在同一个仓库中，版本控制更加简单。
- **统一构建与部署**：可以统一管理构建和部署流程，减少配置复杂性。
- **团队协作高效**：所有代码在一个仓库中，团队成员可以更方便地协作。

**缺点**：

- **仓库体积大**：随着项目数量增加，仓库体积会变得庞大，克隆和拉取速度变慢。
- **构建性能问题**：如果构建工具不支持增量构建，整体构建时间可能会较长。
- **权限管理复杂**：在大型团队中，如何管理不同项目的访问权限是一个挑战。
- **学习曲线高**：需要掌握 Monorepo 工具（如 Lerna、Nx）的使用，增加了学习成本。
